Read this first. It orients you (Claude Code) to the codebase so you can continue the
work. The backlog is in `ROADMAP.md`.

## What this is
Care Royal is a **multi-tenant SaaS sold to home-care agencies**. Each agency is a
`tenant`. It runs the agency end to end: family bookings, caregiver scheduling &
clock-in, e-sign care documents, a document studio (paystubs/invoices/receipts/letters/
care plans), payments, payroll, leads, recruiting, reviews, and reporting.

Business model (do not violate — see also owner memory):
- Agencies **self-connect their own** Stripe (payments) and Gusto *or* use Care Royal's
  in-app pay documents. Care Royal **never holds funds, files taxes, or requires any
  license.** Each agency uploads its own leads.
- Clients don't self-signup — they **Request a quote** at `/quote` (public form) which
  lands in the agency's pipeline. Caregivers apply at `/apply`. Each agency has a public
  microsite at `/care?a=CODE`.

## Stack & architecture
- **Next.js 14 static export** (`output: "export"`, `trailingSlash: true`) + React 18 +
  TypeScript + Tailwind. Deployed as static files to Cloudflare Pages (prod) / GitHub
  Pages (demo).
- **Firebase is the ONLY backend.** Auth (email/password) + **client-side Firestore**.
  There is no REST server. `app/lib/fb.ts` implements a GET/POST "endpoint contract"
  directly against Firestore; security is enforced by `firestore.rules`.
- **Transactional email** runs in **`functions/api/email.js`** — a Cloudflare Pages
  Function (Gmail API + domain-wide-delegated service account, same technique as
  Tegula; JWT signed with Web Crypto). Secret `GMAIL_SERVICE_ACCOUNT` set in the
  Cloudflare Pages dashboard. See `EMAIL.md`. We do NOT use Firebase Functions for
  email — the Firebase project is on Spark (no Blaze billing).
- Other server-only work (Stripe, future AI/Twilio) still lives in **`cloud-functions/`**
  (Firebase Cloud Functions); those need the Blaze plan and are not deployed.

## Key files
- `app/lib/fb.ts` — Firestore data layer (real mode). The source of truth for the API.
- `app/lib/demo.ts` — in-browser mock (demo mode). **Mirror every fb.ts endpoint here.**
- `app/lib/session.ts` — Firebase auth + `apiGet`/`apiPost`/`publicPost`; `signUp`.
- `app/lib/tax.ts` — 2025 federal + CA payroll tax engine (paystubs).
- `app/lib/firebase.ts` — public Firebase web config.
- `firestore.rules` — tenant isolation + role gating + public collections.
- `app/{agency,caregiver,family,owner}/page.tsx` — the four portals.
- `app/{quote,apply,care,login}/page.tsx` — public pages.
- `components/` — PortalShell (shell+nav+notification bell), Icon, Charts, Drawer,
  CalendarView, MessagesPanel, DocumentsPanel, SignaturePad, DocStudio.
- `cloud-functions/index.js` — Stripe Connect + Gmail transactional email.
- Docs: `SETUP.md`, `PAYMENTS.md`, `EMAIL.md`, `DATAMODEL.md`, `ROADMAP.md`.

## How to add a feature (the pattern)
1. Add the endpoint to `app/lib/fb.ts` (real Firestore).
2. Mirror it in `app/lib/demo.ts` (localStorage mock) so demo mode stays whole.
3. Add/adjust `firestore.rules`; deploy: `firebase deploy --only firestore:rules --project care-royale2-4dgwu0`.
4. Build the UI in the relevant page/component using the design-system classes.
5. `npm run build` must stay green. Verify new rules against the live project with REST
   (create/read/negative-case + cleanup) before declaring done.

## Conventions & gotchas
- **No emojis or dingbats** anywhere in UI/copy. Use the line icons in `components/Icon.tsx`.
- Page files (`app/**/page.tsx`) may **only** default-export the component — no other
  named exports (Next build error otherwise). Put shared helpers in `components/` or `lib/`.
- Design system in `app/globals.css`: `card`, `btn-primary/ghost/soft/danger`, `field`,
  `label`, `badge-{brand,ok,gold,warn,danger,muted}`, `chip-on/off`, `hero-gradient`,
  `app-bg`. Fonts: Fraunces (serif headings) + Inter. Brand indigo `#4B39EF`, teal `#39D2C0`.
- Demo login: `grigoryan` / `201816` → all portals; demo join code `DEMO24`.
- PDFs are generated by styled-HTML print windows (no jsPDF dep) — see `printDoc` and
  `DocStudio`.

## Run & deploy
- Dev: `npm run dev` → http://localhost:3000
- Build: `npm run build` → `./out`
- Rules: `firebase deploy --only firestore:rules --project care-royale2-4dgwu0`
- Functions (needs Blaze + secrets): `firebase deploy --only functions` (see PAYMENTS.md, EMAIL.md)
- Demo auto-deploys to GitHub Pages on push to `origin/main` (`.github/workflows/pages.yml`).
- Remotes: `origin` = care-royal-web (dev + demo), `prod` = TheCareRoyal (live domain — push only when asked).
